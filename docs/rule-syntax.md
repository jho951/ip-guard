# Rule Syntax

`ip-guard-core`는 규칙 원문을 줄 단위로 파싱합니다. 각 줄은 하나의 규칙이어야 하며, 빈 줄은 무시됩니다.

`#` 또는 `//` 뒤의 내용은 주석으로 처리됩니다.

```text
127.0.0.1      # localhost
192.168.0.0/16 // private subnet
```

## 지원 문법

| 문법 | IPv4 | IPv6 | 예시 |
| --- | --- | --- | --- |
| Single | 지원 | 지원 | `127.0.0.1`, `2001:db8::1` |
| CIDR | 지원 | 지원 | `192.168.0.0/16`, `2001:db8::/32` |
| Range | 지원 | 지원 | `192.168.0.1-192.168.0.255`, `2001:db8::1-2001:db8::f` |
| Wildcard | 지원 | 미지원 | `192.168.*.*` |

상위 플랫폼 모듈이 `ip-guard`를 감싸는 경우에도 이 문법은 축소 없이 전달되어야 합니다. 소비자 애플리케이션이 플랫폼 facade에 raw rule 목록을 넘기고, 플랫폼 내부에서 `IpGuardEngine`을 생성하는 구조를 권장합니다.

## Single

단일 IP 주소와 정확히 일치합니다.

```text
127.0.0.1
2001:db8::1
```

## CIDR

주소와 prefix를 `/`로 구분합니다. prefix 범위는 IP 패밀리에 맞아야 합니다.

```text
192.168.0.0/16
2001:db8::/32
```

IPv4는 `0`부터 `32`, IPv6는 `0`부터 `128`까지의 prefix를 사용합니다.

## Range

시작 IP와 끝 IP를 `-`로 구분합니다.

```text
192.168.0.1-192.168.0.255
2001:db8::1-2001:db8::f
```

시작 IP와 끝 IP는 같은 패밀리여야 합니다. IPv4와 IPv6를 섞은 range는 거부됩니다.

## IPv4 Wildcard

IPv4 wildcard는 고정된 앞쪽 octet 뒤에 연속된 `*`를 붙이는 문법입니다. 내부적으로 CIDR 규칙으로 변환됩니다.

```text
10.*.*.*
192.168.*.*
```

허용되는 wildcard:

```text
10.*.*.*
192.168.*.*
192.168.1.*
```

거부되는 wildcard:

```text
*.*.*.*
10.*
10.*.*
192.168.*
10.*.1.*
192.*.1.*
300.1.*.*
```

제약:

- wildcard는 IPv4에서만 지원됩니다.
- wildcard도 IPv4 주소처럼 4개 octet을 모두 작성해야 합니다.
- `*`는 뒤쪽에 연속해서만 올 수 있습니다.
- 모든 octet이 `*`인 규칙은 너무 넓어서 거부됩니다.
- 숫자 octet은 `0`부터 `255` 사이여야 합니다.

## 입력 IP 정규화

요청 IP는 `IpParser` 기준으로 정규화됩니다.

```text
[2001:db8::1]:443
fe80::1%en0
127.0.0.1:8080
```

`[IPv6]:port` 형식은 대괄호 안의 IPv6 주소를 사용합니다. IPv6 zone 표기는 `%` 뒤를 제거합니다. IPv4 `host:port` 형식은 host 부분만 사용합니다.

## 파싱 순서

규칙은 다음 순서로 해석됩니다.

1. IPv4 wildcard
2. CIDR
3. Range
4. Single

따라서 `*`가 포함된 IPv4 형식은 먼저 wildcard로 검사되고, `/`가 있으면 CIDR로, `-`가 있으면 range로 처리됩니다.
